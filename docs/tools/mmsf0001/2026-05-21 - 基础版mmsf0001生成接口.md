## 这个接口是干什么的

这个接口主要是用来处理微信 iPad 协议登录里的风险验证参数。

现在的新版本登录里，很多时候不是扫完码就完事了。服务端会继续要求过图片验证码，过完之后新设备场景下还可能继续下发 PIN 校验。

问题在于，这一段验证流程里用到的关键参数，普通浏览器直接打开页面是跑不通的。只靠网页本身，也拿不到后面请求真正需要的安全参数。

根据现有研究，这里有一类和客户端环境强相关的参数，也就是 `mmsf0001 / securityInfo`。它不是普通网页参数，也不是简单把登录上下文改一改就能拼出来的东西。

所以这个接口主要是为了解决下面这件事：

- 微信 iPad 协议登录过程中，遇到风险验证时，后续请求依赖额外的安全参数
- 这个安全参数不是普通浏览器自己能拿到的
- 如果这块能力没有补齐，后面的验证码、PIN、风险确认流程就很难继续联调
- 所以它本质上是在补登录风控链路里缺的那一段能力

## 它最终是为了解决什么事情

围绕当前项目，这个接口最终是为了解决：

- 微信 iPad 协议登录时，风险验证阶段缺少关键安全参数，导致验证流程走不下去的问题

更直接一点说，就是下面这个场景：

- 登录过程中已经拿到了二维码上下文
- 服务端又要求继续做图片验证码或新的设备安全确认
- 但后续请求依赖一段客户端侧的风险参数
- 如果没有这段参数，就算验证码页面拿到了，也很难把整条链路接下去

所以这个接口解决的不是“生成一个字符串”这么简单，而是：

- 风险验证请求有没有前置参数可用
- 验证码和 PIN 相关流程能不能继续往下接
- 安全校验这一段能不能从只会分析，变成可以实际联调

## 研究思路

这个方向的研究思路，主要是先把问题拆开看：

1. 先确认验证码页面为什么不能直接在普通浏览器里完成
2. 再确认微信客户端里到底额外带了什么安全参数
3. 再看这段参数和设备环境、登录态、证书材料是不是绑在一起
4. 最后把这块能力单独整理成接口，方便独立联调和验证

按照现在的研究结果，这一段能力通常不是网页自己算，而是需要结合客户端安全上下文来准备。公开层面可以把它理解成：

```text
登录上下文 + 设备环境 + 证书材料 + 安全封装 -> 风险验证参数
```

这里面会涉及客户端侧安全链路、证书材料准备和传输层上下文配合。对外只展示研究方向，不公开最关键的细节。

## 现在已经能做到什么

现阶段，已经能把这块能力整理成可独立调用的接口方向，用来支撑风险验证链路研究：

- 可以区分上下文模式和本地参数模式
- 可以把风险参数准备过程单独抽出来做联调
- 可以对比不同设备和上下文对结果的影响
- 可以把证书材料准备逻辑统一起来
- 可以给后面的风险验证请求提供基础输入

简单说，现在不是只知道“这里有个参数”，而是已经能往前走到：

- 可以单独调用
- 可以单独展示
- 可以接进后续验证流程继续研究

## 公开示意

为了让外部能看懂这里到底做了什么，下面放一个简化示意。

这里展示的是结构和调用方式，不是可直接复现的核心实现。

### 伪代码示意

```go
type BuildSecurityInfoReq struct {
    Wxid         string
    Uuid         string
    DeviceType   string
    ClientVer    uint32
    CertVersion  uint32
    ContextMode  bool
}

type BuildSecurityInfoResp struct {
    Success      bool
    SecurityInfo string
    ContextSource string
    Warnings     []string
}

func BuildSecurityInfo(req BuildSecurityInfoReq) (*BuildSecurityInfoResp, error) {
    // 1. 读取登录或临时上下文
    // 2. 补齐设备环境
    // 3. 选择证书材料
    // 4. 组织安全载荷
    // 5. 输出后续验证链路使用的 securityInfo
    return &BuildSecurityInfoResp{}, nil
}
```

这个示意想表达的重点很简单：

- 输入不只是一个字符串
- 里面会带上登录上下文、设备信息和证书版本
- 输出的是给后续风险验证流程使用的安全参数

### 返回格式示意

项目里这类接口对外可以整理成类似下面这种返回结构：

```json
{
  "Code": 0,
  "Success": true,
  "Message": "success",
  "Data": {
    "contextSource": "temp.uuid",
    "securityInfo": "BASE64_xxxxx",
    "securityInfoUrlEncoded": "URLENCODE_xxxxx",
    "usedParams": {
      "deviceType": "ipad",
      "clientVersion": 123456789,
      "rsaVersion": 381
    },
    "warnings": [
      "cert loaded from cache",
      "context repaired from login state"
    ]
  }
}
```

这类返回值的意义是：

- `securityInfo` 是后续风险验证请求真正会用到的核心输出
- `contextSource` 可以看出这次走的是哪种上下文来源
- `usedParams` 方便分析本次结果受哪些条件影响
- `warnings` 用来观察有没有缺省值、回补值或者证书切换

### PB 结构的大致方向

从公开角度讲，这一段数据大致可以理解成两层：

```text
外层:
  客户端版本
  证书版本
  设备类型
  密文字段

内层:
  设备相关字段
  登录态相关字段
  风险验证相关载荷
```

不公开真实字段号和真实编码细节，但从研究方向看，它不是普通单层参数。

## 关于 381 key

这里说的 `381 key`，可以理解成这一类风险参数在封装时需要用到的证书材料版本。

大白话说就是：

- 你前面的设备信息、登录上下文都准备好了
- 但如果没有对应的证书材料，最后这段安全参数还是很难封出来
- 所以 `381` 这类东西，本质上是这条链路里的“封装材料”

从研究意义上看，它主要解决的是：

- 没有证书时怎么补齐材料
- 有多个版本时怎么选可用版本
- 结果生成时怎么保持版本一致性
- 后续联调时怎么减少因为证书问题带来的不稳定

公开层面可以把它和 `mmsf0001` 的关系理解成：

```text
登录上下文 + 设备环境 + 381 证书材料 -> 安全封装 -> securityInfo
```

更进一步说，这一段能力通常不会只停留在业务参数层，还会和客户端安全传输环境配合。也正因为这样，这类风险参数不是普通浏览器环境就能随便补出来的。

## 它在业务上能带来什么效果

从实际效果看，这个方向的价值主要体现在下面几个方面：

- 遇到登录风险验证时，不再只能停留在页面层分析，而是能继续研究后续请求所需的安全参数
- 可以把原本零散、依赖上下文的风险参数准备过程，整理成更稳定的能力模块
- 便于继续推进验证码、验证页、风控接口这些后续环节的联调工作
- 让微信 iPad 协议登录研究从“能发起登录”进一步推进到“能处理复杂验证分支”

对于合作方来说，这类能力的实际意义很直接：

- 能否补齐登录过程中的关键风控参数
- 能否继续推进需要安全验证的登录链路
- 能否把原本难以整理的风险验证前置能力沉淀成可对接模块

## 适用用途
